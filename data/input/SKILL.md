# 【机构间REITs项目整理】技能

## 功能
自动爬取上交所和深交所持有型ABS/机构间不动产项目，筛选2023年10月后的申报项目并汇总

## 核心特性

✅ **上交所全自动爬取**
- 自动翻页，无页数限制
- 智能停止：检测到受理日期早于2023-10-01自动停止翻页
- 筛选条件：债券名称含有"持有型"或"机构间"

✅ **深交所自动下载**
- 自动下载官方Excel数据
- 自动解析并筛选符合条件的项目
- 备注：云服务器IP可能被拦截，建议本地运行

✅ **数据自动处理**
- 自动合并两个交易所数据
- 自动筛选受理日期≥2023-10-01的项目
- 自动标准化字段名称
- 按受理日期降序排序

## 使用方法

### 一键运行（推荐）
```bash
cd ~/.openclaw/workspace/skills/机构间REITs项目整理/
python3 main.py
```

脚本会自动完成：
1. 爬取上交所所有符合条件的项目（自动翻页直到日期早于2023-10-01）
2. 自动下载深交所Excel并筛选
3. 合并数据并导出Excel

### 首次使用需安装依赖
```bash
pip install -r requirements.txt
playwright install chromium
```

## 筛选规则

### 关键词筛选
债券名称包含以下任一关键词：
- "持有型"
- "机构间"

### 时间筛选
受理日期 ≥ 2023年10月1日

## 输出字段

| 字段名 | 说明 |
|--------|------|
| 交易所 | 上交所/深交所 |
| 债券名称 | 项目全称 |
| 计划管理人 | 管理人名称 |
| 品种 | 资产支持证券（ABS）/私募/小公募等 |
| 拟发行金额(亿元) | 申报发行规模 |
| 项目状态 | 已受理/已反馈/通过/终止等 |
| 更新日期 | 最新状态更新日期 |
| 受理日期 | 交易所受理日期 |

## 输出文件

`持有型ABS申报项目汇总_YYYYMMDD.xlsx`（保存在/root/.openclaw/workspace/skills/机构间REITs项目整理/output/）

## 技术说明

### 上交所爬取策略
- 使用playwright浏览器自动化
- 逐页爬取，每页等待2秒避免请求过快
- 智能停止机制：当前页所有项目受理日期都早于2023-10-01时停止翻页
- 无头模式运行，适合云服务器环境

### 深交所下载策略
- 直接调用官方Excel下载API
- 尝试多个备用下载地址
- 自动识别表头格式并标准化字段名

## 故障排查

### 上交所爬取失败
- 检查playwright是否正确安装：`playwright install chromium`
- 检查网络连接是否正常
- 页面结构可能变化，需要更新选择器

### 深交所下载失败
- 云服务器IP可能被拦截，建议本地运行
- 或手动下载Excel放到skill目录，重命名为`szse_data.xlsx`后运行

### 运行超时
- 上交所翻页较多时可能需要较长时间
- 建议使用`timeout 300 python3 main.py`设置5分钟超时

## 核心工作流&模型分配规则
### 运行模式区分
| 模式 | 定位 | Token消耗 | 适用场景 |
|------|------|-----------|----------|
| 基础爬取模式 | 仅爬取+导出Excel | 0 Tokens | 日常更新数据、获取原始清单 |
| 深度分析模式 | 爬取+清洗+专业分析 | 1000~15000 Tokens | 出具投资分析报告、项目估值、配置建议 |

### 全环节模型分配
| 阶段 | 处理难度 | 主模型 | 次模型（交叉校验） | Token消耗预估 |
|------|----------|--------|--------------------|---------------|
| 数据爬取/清洗/导出 | 无（纯代码逻辑） | - | - | 0 |
| 基础统计、普通通知、结果导出 | 简单 | 豆包 | - | 1000~2000 Tokens |
| 项目估值、深度分析、专业报告撰写 | 复杂 | Claude Opus | GPT-4o | 5000~12000 Tokens |

## 异常处理全流程规则
### 处理优先级（严格执行）
`代码规则处理 > 历史错误库匹配 > 小模型处理 > 大模型兜底`，90%常见错误实现零Token处理。

| 流程 | 处理逻辑 | Token消耗 |
|------|----------|-----------|
| 1. 错误捕获 | 程序自动捕获异常、采集错误日志和栈信息 | 0 |
| 2. 规则匹配 | 优先匹配历史错误库，自动处理常见问题：<br>• 网络超时 → 自动重试3次<br>• 页面加载超时 → 延长等待时间<br>• 元素定位失败 → 自动尝试3种降级定位逻辑<br>• 已知反爬拦截 → 自动切换User-Agent | 0 |
| 3. 错误分类 | 未知错误调用豆包识别类型（代码逻辑/页面结构升级/反爬升级/接口变更） | <500 Tokens |
| 4. 方案生成 | 优先调用豆包生成修复方案，方案无效再升级大模型 | 1000~3000 Tokens |
| 5. 代码修改 | 优先调用豆包生成修复代码，连续失败2次升级大模型 | 2000~5000 Tokens |
| 6. 回归测试 | 自动运行测试用例验证修复结果 | 0 |
| 7. 错误入库 | 新错误+解决方案存入历史错误库，下次同类错误直接匹配 | 0 |
| 8. 通知反馈 | 精简结果通知用户 | <300 Tokens |

## 自动降级&Token最优化策略
### 自动降级触发条件（仅满足以下条件才升级到大模型）
1. 豆包输出结果置信度<80%
2. 豆包连续2次输出无效方案/代码
3. 错误类型明确为「页面重构/反爬重大升级/接口完全变更」等高复杂度场景

### Token优化规则
1. 上下文复用：同一错误全流程复用分析上下文，每次调用仅增量传递新信息，减少Token消耗30%
2. 批量处理：同类错误批量分析，避免重复调用模型
3. 严格限制上下文长度：错误分析仅传递关键错误日志和代码片段，禁止传递全量无关内容

### 错误处理成本分级
| 错误等级 | 总Token消耗范围 | 处理耗时 |
|---------|----------------|----------|
| 低（常见错误） | 0~800 Tokens | <5分钟 |
| 中（普通页面/接口调整） | 1500~3500 Tokens | 5~20分钟 |
| 高（重大重构/反爬升级） | 6000~12000 Tokens | 20~60分钟 |
